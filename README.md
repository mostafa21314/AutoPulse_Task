# AutoPulse

AutoPulse Intern Challenge — a Flutter app that reads a vehicle's service
records with OCR (Google Gemini) and produces a predictive maintenance
schedule, exportable as a PDF deliverable. Built and tested against a Jeep
Grand Cherokee service history.

---

## Workflow

The app walks the user through five stages:

1. **Upload** — the user uploads a maintenance record (JPG/PNG/PDF/HEIC) or
   captures one with the camera. ([upload screen in `main.dart`](lib/main.dart))
2. **Preview** — the picked file is shown for confirmation, then sent off for
   analysis. ([`preview_screen.dart`](lib/screens/preview_screen.dart))
3. **Analyze (OCR)** — the document bytes are sent to **Gemini**, which returns
   the structured service history: the vehicle identity, every service visit,
   and a canonical list of which maintenance components each visit covered.
   ([`ocr_service.dart`](lib/services/ocr_service.dart))
4. **Review & edit** — the extracted history is displayed. The user can fix any
   wrongly-read row or add a missed one; added rows slot into chronological
   order automatically. ([`results_screen.dart`](lib/screens/results_screen.dart))
5. **Schedule** — the user enters a driving profile as explicit choices
   (overspeeding y/n · urban vs off-road · smooth vs sudden braking). From here:
   - **Gemini summary** — a forward-looking schedule is generated by Gemini for
     on-screen display. ([`schedule_service.dart`](lib/services/schedule_service.dart),
     [`maintenance_schedule_screen.dart`](lib/screens/maintenance_schedule_screen.dart))
   - **PDF deliverable** — a PDF is generated from **hard-coded modelling
     logic** (no AI): part 1 explains the modelling and how each driving
     variable was accounted for; part 2 is the adjusted maintenance schedule for
     the next 80,000 km. ([`maintenance_model.dart`](lib/services/maintenance_model.dart),
     [`schedule_pdf_service.dart`](lib/services/schedule_pdf_service.dart))

### Modelling logic (PDF)

For each tracked component:

- `baseline_interval` = manufacturer-recommended km.
- Each driving choice contributes a **per-component multiplier** (e.g.
  overspeeding shortens engine-oil life ×0.7; smooth braking extends brake-pad
  life ×1.33).
- `adjusted_interval = baseline × (product of applicable factor multipliers)`,
  rounded to the nearest 500 km.

Scheduling per component:

- `last_done` is cross-validated against the history — we walk the reports
  newest-first and take the most recent visit that actually serviced that
  component (matched on Gemini's component tags, with keyword fallback).
- `next_due = last_done + adjusted_interval`.
- `current_odo > next_due` → **OVERDUE**; within 1,000 km → **DUE SOON**;
  otherwise **SCHEDULED**.

## Project structure

### Entry point
- **`lib/main.dart`** — app bootstrap (loads `.env`), the navigation shell, and
  the Upload screen with file-picker / camera capture.
- **`lib/theme.dart`** — brand colors and text styles.

### Models (`lib/models/`)
- **`service_record.dart`** — `VehicleIdentity`, `ServiceInstant` (one visit,
  including the canonical `components` tagged by Gemini), and `ServiceRecord`.
  JSON (de)serialization plus sample placeholder data.
- **`driving_profile.dart`** — the user's driving choices (overspeeding,
  off-road, sudden braking).
- **`maintenance_schedule.dart`** — `MaintenanceSchedule` / `MaintenanceItem`
  for the Gemini-generated on-screen summary.

### Screens (`lib/screens/`)
- **`preview_screen.dart`** — confirm the file, then trigger OCR analysis.
- **`results_screen.dart`** — editable service history; the driving-profile
  choices sheet; launches schedule generation.
- **`maintenance_schedule_screen.dart`** — the Gemini schedule summary plus the
  **Generate Schedule PDF** action.

### Services (`lib/services/`)
- **`gemini_client.dart`** — thin shared wrapper over the Gemini
  `generateContent` REST endpoint; runs the model in JSON mode and returns the
  parsed object. Used by both calls below.
- **`ocr_service.dart`** — document → `ServiceRecord` (the OCR prompt).
- **`schedule_service.dart`** — history + profile → Gemini `MaintenanceSchedule`
  (the on-screen summary).
- **`maintenance_model.dart`** — the hard-coded modelling + scheduling logic for
  the PDF (components, per-factor multipliers, cross-validation, due-status).
- **`schedule_pdf_service.dart`** — builds the two-part PDF document.
- **`pdf_saver.dart` / `pdf_saver_web.dart` / `pdf_saver_io.dart`** — saving the
  PDF, split by platform via conditional import (direct browser download on web,
  OS share sheet on desktop/mobile).

### Widgets (`lib/widgets/`)
- **`app_header.dart`**, **`buttons.dart`**, **`floating_nav_bar.dart`**,
  **`service_field_card.dart`** — shared UI building blocks.

## Setup

1. Install Flutter (and add it to your PATH).
2. Copy the env template and add your key:
   ```
   cp .env.example .env
   # then set GEMINI_API="your-key"   (get one at https://aistudio.google.com/apikey)
   ```
3. Fetch packages and run:
   ```
   flutter pub get
   flutter run -d chrome
   ```

The Gemini model is configured in [`gemini_client.dart`](lib/services/gemini_client.dart)
(`gemini-3.1-flash-lite`).

## Architecture note — frontend-only by design

For this challenge, **everything runs in the Flutter frontend**, including the
direct calls to the Gemini API (with the key loaded from `.env`). This keeps the
project simple and self-contained, with no server to deploy.

**This is not how a production system should be built.** Shipping an API key in
a client bundle exposes it — anyone can extract it from the app and run up your
quota — and it makes the AI behavior impossible to govern centrally. A
production-grade architecture would put a **backend service** between the app
and the model provider:

- The app uploads the document to the backend; the **API key lives only on the
  server**, never in the client.
- The backend handles communication with the model (OCR + analysis), and can add
  auth, rate limiting, retries, caching, validation, logging, and cost controls.
- The modelling/scheduling logic would also live server-side (or in a shared
  service) so it can be versioned and audited independently of app releases.

The current design is a deliberate, documented trade-off for speed of
delivery.
